# v0.3 Plan — Frontend Local "Import comments" Demo Panel (Phase 7)

> **Status: ✅ implemented (Phase 7).** This describes the frontend panel that lets a
> user paste **fixture-shaped** GitHub/GitLab comment JSON, normalize it via the
> existing local-only `POST /api/import-comments`, preview the result, and load it
> into the existing `commentThreads` state so suggested replies run on imported data.
>
> Shipped: `frontend/src/types/gitImport.ts`, `frontend/src/api/importComments.ts`
> (`importComments` + `ImportApiError`), `frontend/src/components/ImportCommentsPanel.tsx`
> (+ tests), and `DiffInputPanel` integration that owns `importedThreads`, previews
> them in an "Imported comments" group (remove / clear), and merges + dedupes them
> with manual threads (`[...importedThreads, ...manualThreads]`) when building the
> request. Imported threads are read-only this phase (inline editing deferred).
>
> Hard constraints (must hold in the implementation): **no** GitHub/GitLab URL input,
> **no** OAuth, **no** token input, **no** live provider API calls, **no** posting,
> **no** real AI/LLM, and **no** claim that this is live GitHub/GitLab integration.
> Backend reference: [`v0.3-plan-git-comment-import-mappers.md`](v0.3-plan-git-comment-import-mappers.md),
> [`future-git-provider-comment-import.md`](future-git-provider-comment-import.md).

## 1. Product goal

Give the demo a visible, honest **normalization boundary**: a user pastes
provider-shaped comment JSON (the same synthetic shape the backend fixtures use),
clicks **Normalize comments**, sees how many threads were produced (plus any
warnings) and a preview, then clicks **Load imported threads** to feed them into the
existing local `commentThreads`. The existing review flow then generates suggested
replies on the imported data — proving the import → normalize → review/reply pipeline
end to end, entirely locally.

## 2. Non-goals

- **No URL input / no fetching.** The panel only accepts pasted JSON; it never takes
  a PR/MR URL and never calls GitHub/GitLab.
- **No OAuth, no token field.** Nothing token-like appears anywhere in the UI.
- **No posting.** Imported threads are input only; suggested replies stay copy-only.
- **No real AI/LLM.** Reply generation remains the existing deterministic flow.
- **No new review behavior.** Imported threads are just `CommentThread`s; detection,
  routing, tone, risk, and recommendation are unchanged (backend invariance is
  already proven).
- **No misleading framing.** Copy explicitly says *local, fixture-based demo*.
- **Not a replacement** for the manual `CommentThreadsInput` — both coexist.
- **Full inline editing of imported multi-comment threads is out of scope** for this
  phase (see §8 state model / §13 risks).

## 3. User flow

1. In the **Merge request** panel, the user expands **"Import comments (local
   demo)"** (a collapsible panel near `CommentThreadsInput`).
2. The user selects a **provider** (`github` / `gitlab`) and a **source**:
   - GitHub → `github_review_comments` or `github_issue_comments`
   - GitLab → `gitlab_discussions`
   (The source list is filtered by the selected provider.)
3. The user pastes a **JSON payload** (array or object) into a textarea. A
   "Load example" affordance can drop in a small synthetic sample for convenience.
4. The user clicks **Normalize comments**.
   - The frontend first **validates the JSON locally** (`JSON.parse`). On failure it
     shows an inline error and does **not** call the API.
   - On success it `POST`s `{ provider, source, rawPayload }` to
     `/api/import-comments`.
5. The frontend shows the result:
   - **imported thread count** (e.g. "4 threads normalized"),
   - **warnings** (if any), each rendered as a non-blocking notice,
   - a **read-only preview** of the normalized threads (file/line, status, comment
     count, first comment snippet, provenance like discussion/comment id).
6. The user clicks **Load imported threads**.
   - The normalized `CommentThread`s populate the panel's imported-threads state
     (owned by `DiffInputPanel`), shown as a labeled "Imported" group.
7. The user clicks **Run Review** as usual; the request includes the imported threads
   (merged with any manual threads), and suggested replies are generated normally.

Empty/edge results are handled honestly (see §9).

## 4. UI placement

- A new collapsible `<details>` panel titled **"Import comments (local demo)"**,
  placed **directly above or below `CommentThreadsInput`** inside `DiffInputPanel`,
  visually grouped with comment-thread input.
- A short, prominent helper line: *"Local demo — paste provider-shaped JSON to
  normalize it into comment threads. Nothing is fetched from GitHub/GitLab, no tokens
  are used, and no comments are posted."*
- Reuses existing styles (`.threads-panel`, `.input`, `.button`, `.link-button`,

- **`ImportCommentsPanel.tsx`** *(new)* — self-contained panel that owns its local UI

## 6. TypeScript types needed (`frontend/src/types/gitImport.ts`, new)

Mirror the backend contract (camelCase), reusing the existing `CommentThread` type:

```ts
import type { CommentThread } from "./review";

export type GitProviderType = "github" | "gitlab";

export type ImportSource =
  | "github_review_comments"
  | "github_issue_comments"
  | "gitlab_discussions";

export interface ExternalCommentReference {
  provider: GitProviderType;
  repository?: string | null;
  projectId?: string | null;
  pullRequestNumber?: number | null;
  mergeRequestIid?: number | null;
  discussionId?: string | null;
  reviewId?: string | null;
  commentId?: string | null;
  noteId?: string | null;
  webUrl?: string | null;
  isOutdated?: boolean | null;
}

export interface ImportedCommentThread {
  thread: CommentThread;
  externalReference: ExternalCommentReference;
  warnings: string[];
}

export interface ImportCommentsRequest {
  provider: GitProviderType;
  source?: ImportSource | null;
  rawPayload?: unknown; // array or object; validated via JSON.parse first
}

export interface ImportCommentsResponse {
  provider: GitProviderType;
  threads: ImportedCommentThread[];
  warnings: string[];
}
```

A small UI constant maps provider → allowed sources (with human labels) for the
source dropdown, kept in the panel or a tiny `lib` helper.

## 8. State ownership & how imported threads flow into `commentThreads`

**Single source of truth for what gets sent:** `DiffInputPanel` builds
`request.commentThreads` from two clearly-labeled sources and concatenates them:

- `manualThreads` — from `CommentThreadsInput` (unchanged, internal drafts).
- `importedThreads: CommentThread[]` — new state in `DiffInputPanel`, set when
  `ImportCommentsPanel` calls `onLoadThreads`.

On submit:

```ts
const merged = dedupeById([...importedThreads, ...manualThreads]);
if (merged.length > 0) request.commentThreads = merged;
```

`dedupeById` keeps the first occurrence per `thread.id` (imported ids are
deterministic/synthetic, e.g. `github:rc:1001`, so collisions are unlikely and
de-duping is safe).

**Why keep imported threads as full `CommentThread[]` (not flattened into drafts):**
the existing `ThreadDraft` model holds a *single* comment, but imported threads can
have multiple comments (e.g. a GitHub root + reply, or a GitLab discussion with
several notes). Flattening would lose comments that matter for routing/preview, so
imported threads are stored faithfully and shown in a **read-only "Imported" preview**
with a **Remove** (per thread) and **Clear imported** affordance — manageable, but not
inline-editable this phase. Manual editing remains available via
`CommentThreadsInput`. This satisfies "avoid duplicating comment-thread state" (there
is exactly one merged list sent) while preserving fidelity.

*(Alternative considered: lift `CommentThreadsInput`'s drafts into `DiffInputPanel`
and convert imported threads into editable single-comment drafts. Rejected for this
phase because it is lossy for multi-comment threads and a larger refactor; revisit if
inline editing of imported threads becomes a requirement.)*

## 9. Warning / error handling & empty states

- **Invalid JSON (client-side):** inline error ("That doesn't look like valid
  JSON."), **no** API call, Normalize button effectively a no-op until fixed.
- **API 400 (unsupported/ambiguous source):** show the backend `detail` message as an
  error notice (the provider/source selectors make this rare, but a mismatch or empty
  source still surfaces clearly).
- **API/network failure:** friendly message mirroring `ReviewApiError` ("Could not
  reach the import service. Is the backend running on port 8000?").
- **200 with `threads: []` + warnings:** show "No comment threads were produced from
  this payload." plus the returned warning(s); **Load imported threads** is disabled.
- **200 with threads:** show count, any warnings (e.g. "reply … references missing
  root …"), and the preview; enable **Load imported threads**.
- **After loading:** show "Loaded N imported threads." and reflect them in the
  Imported group; loading again replaces/append per a documented rule (recommend
  **replace** to avoid silent duplication, with the merged-on-submit dedupe as a
  backstop).

## 10. Test strategy (Vitest + RTL)

`ImportCommentsPanel.test.tsx` and an update to `DiffInputPanel.test.tsx`:

1. **Renders provider/source controls** and the JSON textarea; source options change
   with provider (GitHub shows two sources, GitLab one).
2. **Rejects invalid JSON before calling the API** — mock `importComments`; type
   non-JSON, click Normalize, assert an error shows and the API mock was **not**
   called.
3. **Calls the API with a camelCase request** — valid JSON + selections → assert the
   client was called with `{ provider, source, rawPayload }` (rawPayload parsed).
4. **Displays warnings/errors** — mock a response with `warnings`; assert they render;
   mock a 400 `ImportApiError`; assert the detail message renders.
5. **Previews imported threads** — mock a response with two threads; assert count and
   per-thread preview (file/line, comment snippet) render.
6. **Loads imported threads into `commentThreads`** — click Load; assert `onLoadThreads`
   fires with the normalized threads.
7. **Review request includes loaded imported threads** — in `DiffInputPanel.test.tsx`,
   load imported threads, run review, assert `onRun` payload's `commentThreads`
   contains them (merged with manual ones).
8. **No token/URL input exists** — assert there is no input whose label/placeholder
   mentions token/URL; assert the request body has no `token`/`url` keys.
9. **Empty-state** — mock `threads: []` + warning; assert the empty message shows and
   Load is disabled.

Use a typed mock of `api/importComments` (like existing tests mock `api/reviews`).
Reuse `frontend/src/test/fixtures.ts` for a small synthetic `ImportCommentsResponse`.

## 14. Risks & mitigations

- **Misleading "real integration" impression.** *Mitigation:* explicit "local demo /
  fixture-based / no fetch / no tokens / no posting" copy in the panel, README, and
  demo script; the word "paste" everywhere; no URL field.
- **Multi-comment imported threads vs. single-comment draft model.** *Mitigation:*
  store imported threads faithfully as `CommentThread[]` and preview read-only;
  document inline editing as deferred (§8).
- **Duplicate threads when loading twice or overlapping manual ids.** *Mitigation:*
  "load replaces imported set" + `dedupeById` on submit.
- **Large/sensitive pasted payloads.** *Mitigation:* it's local-only and never
  persisted; optionally cap textarea size and show a gentle note; never log payloads.
- **Backend not running.** *Mitigation:* friendly error mirroring `ReviewApiError`.
- **Scope creep toward live import.** *Mitigation:* keep this strictly a normalization
  demo; live fetching/auth stays in the deferred design docs.

## 15. What remains deferred (unchanged by this phase)

Live GitHub/GitLab API calls, OAuth, token input, URL fetching, private-repo support,
auto-posting, inline editing of imported multi-comment threads, and real AI/LLM.
