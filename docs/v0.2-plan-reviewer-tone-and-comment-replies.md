# v0.2 Plan — Reviewer Tone Profiles & Suggested Comment Replies

> **Status: planning only.** Nothing in this document is implemented yet. It
> describes the design for the next iteration on top of the completed,
> local-first MVP. No real AI/LLM calls, no GitHub/GitLab API integration, no
> auto-posting, no auth, and no persistence are introduced by this plan.

## Context

The MVP reviews a diff through multiple personas and returns risk, a merge
recommendation, and per-persona findings (see [`architecture.md`](architecture.md)
and [`review-contract.md`](review-contract.md)). v0.2 adds two capabilities that
sit *on top* of that pipeline without changing how risk is detected:

1. **Reviewer voice/tone profiles** — configurable communication style per
   persona (tone, strictness, verbosity, optional custom instructions).
2. **Suggested replies to existing MR/PR comments** — paste existing comment
   threads and generate *draft* replies from selected reviewer perspectives
   (copy-only; never auto-posted).

## Product goals

- Let users tune *how* a reviewer communicates without changing *what* it flags.
- Make findings feel like they come from a specific kind of reviewer (e.g. a
  supportive mentor vs. a terse staff engineer) to fit team culture.
- Help users respond to review discussions by drafting replies from a chosen
  persona/tone, which they copy and post themselves.
- Keep everything local-first, deterministic, and demoable with the mock
  provider; preserve the existing review contract and UX.

## Non-goals (for v0.2)

- **No real AI/LLM calls.** Tone and replies are rendered by the deterministic
  mock provider (templated). The same `ReviewProvider`/prompt seam is reused so a
  future LLM provider can do this better.
- **No GitHub/GitLab API integration.** Comment threads are pasted/entered
  manually, not fetched.
- **No auto-posting** of replies back to any platform. Output is copy-only
  (clipboard / Markdown).
- **No authentication or persistence.** No accounts, tokens, or stored history.
- **Tone never changes risk detection.** Severity, which findings are produced,
  overall risk, and the merge recommendation are computed from the *raw* analysis
  and are independent of tone/strictness/verbosity.

## User stories

**Tone profiles**
- As a user, I can set a global tone profile (tone, strictness, verbosity) that
  applies to all selected personas.
- As a user, I can override the tone profile per persona (e.g. Security strict,
  Product supportive).
- As a user, I can add optional custom instructions (e.g. "reference our style
  guide") that influence wording only.
- As a user, I can change tone and re-render the *same* findings without
  re-running detection, and the risk/recommendation stay identical.

**Suggested replies**
- As a user, I can paste one or more existing comment threads (with optional file/
  line context) and pick personas + a tone.
- As a user, I get a suggested/draft reply per thread per selected persona.
- As a user, I can copy a reply to the clipboard or export all replies to
  Markdown.
- As a user, it is always clear these are *drafts* I must review and post myself.

## Proposed data models

New models are additive and optional so the existing contract stays
backward-compatible. (Backend Pydantic `CamelModel`; mirrored in
`frontend/src/types/review.ts`.)

```text
ToneStyle      = "direct" | "supportive" | "educational" | "strict" | "curious" | "executive"
Strictness     = "low" | "medium" | "high"
Verbosity      = "brief" | "normal" | "detailed"

ToneProfile {
  tone:               ToneStyle      = "direct"
  strictness:         Strictness     = "medium"
  verbosity:          Verbosity      = "normal"
  customInstructions: str | null     = null   # wording guidance only
}

# Suggested-replies feature
CommentAuthorRole = "author" | "reviewer" | "other"

ThreadComment {
  author:    str
  role:      CommentAuthorRole = "other"
  body:      str
  createdAt: str | null = null   # free-form/ISO; display only
}

CommentThread {
  id:          str
  filePath:    str | null = null
  line:        int | null = null
  comments:    list[ThreadComment]
}

SuggestedReply {
  threadId:   str
  persona:    ReviewerPersona
  tone:       ToneStyle
  replyText:  str
  confidence: float | null = null
}
```

Notes:
- `ReviewFinding` gains no required fields. Tone rendering rewrites the existing
  `title` / `explanation` / `recommendation` strings; the raw severity, file,
  hunk, and confidence are untouched. (Optionally add a non-breaking
  `toneApplied: ToneStyle | null` for transparency/debugging.)

## Backend contract changes

All additive and optional.

- **`ReviewRequest`** gains:
  - `toneProfile: ToneProfile | null` (global default), and/or
  - `personaToneProfiles: dict[ReviewerPersona, ToneProfile] | null` (overrides).
  - Resolution order: per-persona override → global `toneProfile` → built-in
    default. If all omitted, behavior is identical to today.
- **New endpoint — suggested replies:**
  - `POST /api/comment-replies`
  - Request: `{ threads: CommentThread[], selectedPersonas: ReviewerPersona[],
    toneProfile?, personaToneProfiles?, diffText?: string }`
    (`diffText` optional, for extra context).
  - Response: `{ replies: SuggestedReply[] }` (one per thread × selected persona).
- **No change** to risk/recommendation aggregation logic. Tone is applied after
  findings are produced and before serialization; aggregation reads raw
  severities only.

## Frontend UX changes

**Tone profiles**
- A "Reviewer voice" control near the persona selector: tone dropdown, strictness
  and verbosity selectors, and an optional custom-instructions textarea.
- A global tone with optional per-persona overrides (progressive disclosure:
  global first, "customize per persona" expands a small matrix).
- Re-rendering tone on an existing result should be cheap; ideally the client can
  request a re-render without forcing a brand-new detection pass (or simply
  re-submit — detection is deterministic, so the result is stable).
- Finding cards display unchanged structurally; only the wording changes.

**Suggested replies**
- A new "Comment replies" area (tab or section): paste/enter comment threads
  (a simple structured form, or a paste-and-parse helper), pick personas + tone,
  and generate.
- A reply dashboard: grouped by thread, then persona; each reply has a **Copy**
  button and the whole set supports **Export Markdown**.
- Persistent, explicit framing: "Draft replies — review before posting. Nothing
  is posted automatically."

## Provider / mock-provider implications

- **Tone is a rendering layer, not detection.** Introduce a `ToneRenderer`
  (deterministic, templated) that transforms a finding's wording given a
  `ToneProfile`:
  - *tone* → phrasing templates (e.g. supportive adds encouragement; executive
    compresses to outcome + ask; educational adds a short "why").
  - *verbosity* → expand/trim explanation length (brief = one line).
  - *strictness* → framing emphasis of the recommendation ("consider" vs
    "must"), **not** severity.
  - *customInstructions* → mock can only do shallow things (e.g. append a note);
    a real LLM provider would honor it in the prompt.
- **Provider boundary:** the `ReviewProvider` keeps returning `PersonaReview`s
  with raw findings; the engine applies tone rendering after the provider call
  (so mock and a future LLM provider both benefit). Alternatively a provider may
  opt to render tone itself (an LLM would), guarded by a capability flag — to be
  decided in Phase 12/13.
- **Persona registry** (`app/personas/registry.py`) gains tone-aware prompt
  helpers for the future LLM provider; the mock uses templates keyed by
  `(persona, tone, verbosity)`.
- **Suggested replies in mock:** a deterministic reply generator keyed by persona
  focus + thread content + tone (e.g. Security replies about secrets/risk,
  QA about tests). No model, no network.

## Test plan

- **Tone invariance (critical):** for a fixed diff + personas, varying tone /
  strictness / verbosity must **not** change `overallRisk`,
  `mergeRecommendation`, the set/severity of findings, or `findingsBySeverity`.
  Only `title`/`explanation`/`recommendation` strings may differ. Add explicit
  backend tests asserting this invariant across tones.
- **Tone rendering:** each tone/verbosity produces the expected shape (e.g. brief
  = single line; supportive includes supportive phrasing) deterministically.
- **Backward compatibility:** requests without tone fields behave exactly like
  today (snapshot/contract tests).
- **Suggested replies:** `POST /api/comment-replies` returns one reply per thread
  × persona; replies are deterministic; empty threads / no personas handled
  gracefully; invalid persona → 422.
- **Frontend (Vitest + RTL):** tone controls update requests; reply UI renders,
  copy works, Markdown export includes all replies; "draft/copy-only" messaging
  present.
- **Markdown export:** review export optionally notes the tone used; reply export
  is well-formed and grouped by thread/persona.

## Risks / tradeoffs

- **Strictness vs. detection boundary.** Strictness is the most likely place to
  accidentally leak into detection. Mitigation: strictness only affects
  recommendation *framing/wording*; severity thresholds are fixed. Enforce with
  the tone-invariance tests above.
- **Mock tone realism.** Templated tone will feel mechanical compared to an LLM.
  Acceptable for the local-first model; it proves the seam and UX. Set
  expectations in docs (no overclaiming).
- **Reply quality without a model.** Deterministic replies are generic. Framing
  them clearly as *drafts* keeps the human in the loop and avoids overpromising.
- **Contract creep.** Adding tone everywhere risks bloating the API. Mitigation:
  keep all tone fields optional with sane defaults; resolve centrally.
- **Comment-thread input friction.** Manual entry is clunky vs. fetching from a
  platform. Accepted for now; Phase 17 designs import to remove the friction
  later.
- **Auto-posting temptation.** Writing back to real MRs is high-risk (wrong/noisy
  comments, permissions). Explicitly deferred; copy-only keeps v0.2 safe.

## Recommended implementation phases

- **Phase 12 — Reviewer tone profile contract.** ✅ *Done.* Added `ToneProfile` +
  `ToneStyle`/`ToneStrictness`/`ToneVerbosity` enums (backend Pydantic in
  `app/models/tone.py` + TS types), optional `toneProfile` / `personaToneProfiles`
  on `ReviewRequest`, a `resolve_tone_profile(persona, request)` helper, and
  contract + detection-invariance tests (`tests/test_tone_contract.py`). Fully
  backward-compatible and behavior-neutral: no wording is changed yet.
- **Phase 13 — Tone UI + mock tone rendering.** Add the `ToneRenderer`
  (templated, deterministic) applied after findings are produced; build the
  "Reviewer voice" UI (global + per-persona). Add tone-invariance tests.
- **Phase 14 — Existing MR comment thread contract.** Add `CommentThread` /
  `ThreadComment` / `SuggestedReply` models + TS types and a minimal input path
  (structured form or paste-and-parse). No generation yet.
- **Phase 15 — Local suggested reply generation.** Implement the deterministic
  mock reply generator + `POST /api/comment-replies`, honoring persona + tone.
  Tests for determinism and edge cases.
- **Phase 16 — Comment reply dashboard + Markdown export.** Build the reply UI
  (grouped, copy buttons, draft framing) and Markdown export for replies; extend
  review export to note tone where useful.
- **Phase 17 — Future GitHub/GitLab comment import (design only).** Design how
  threads would be imported (building on
  [`future-git-provider-import.md`](future-git-provider-import.md)); still no
  auto-posting.

## Why GitHub/GitLab import and auto-posting come later

- **Prove the local model first.** Tone rendering and reply generation must be
  good and trustworthy with deterministic input before adding the complexity of
  fetching real threads.
- **Security surface.** Import means tokens, scopes, SSRF/rate-limit handling, and
  private-repo concerns — significant work that shouldn't gate the core feature.
- **Auto-posting is high-risk and irreversible-ish.** Posting wrong or noisy
  comments to real MRs damages trust and needs auth, guardrails, and human
  confirmation. Copy-only drafts keep a human in the loop and let us ship value
  safely first. Auto-posting (if ever) would be a deliberate, separate, gated
  phase after import.
