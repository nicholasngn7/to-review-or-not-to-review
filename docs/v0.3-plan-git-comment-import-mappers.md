# v0.3 Plan — Git Provider Comment Import: Contracts & Pure Mappers

> **Status: planning only.** Nothing in this document is implemented yet. It
> describes the *next* implementation phase: provider import **contracts** plus
> **pure, fixture-tested mapper functions** that normalize recorded GitHub/GitLab
> comment JSON into the **existing** `CommentThread` contract.
>
> Scope guardrails for the eventual implementation phase: **no live GitHub/GitLab
> API calls, no OAuth, no token input, no private-repo support, no auto-posting, no
> AI/LLM**. This phase is deliberately **network-free and fixture-only**.
>
> Background design: [`future-git-provider-comment-import.md`](future-git-provider-comment-import.md).
> Contract reference: [`review-contract.md`](review-contract.md).

## 1. Product goal

Build the **foundation** for importing real PR/MR review discussions: a small set of
backend **import contracts** and **pure mapper functions** that convert recorded
provider JSON (checked-in test fixtures) into the existing `list[CommentThread]`
shape the review/suggested-reply pipeline already consumes.

The deliverable is *purely a normalization layer*, fully exercised against fixtures:

```text
recorded provider JSON  -->  pure mapper(...)  -->  ImportedCommentThread[]
                                                     │  .thread  -> existing CommentThread
                                                     └  .external -> ExternalCommentReference (provenance)
```

No endpoint, no network, no UI is required to call it valuable — the mappers are unit
-testable in isolation and prove the normalization rules from the design doc before
any real API/security work begins.

## 2. Non-goals

- **No live network calls.** Mappers take already-parsed JSON (dicts), never URLs or
  HTTP clients.
- **No OAuth, no token input, no private-repo support.** Fixtures stand in for any
  data we'd otherwise fetch; auth is a strictly later, separate phase.
- **No auto-posting and no posting of suggested replies.** Import is read-only;
  replies stay copy-only and `needsHumanReview: true`.
- **No real AI/LLM calls.** Reply generation remains deterministic and local.
- **No new HTTP endpoint required in this phase.** (An optional thin
  `POST /api/import-comments` that calls the mappers on a *fixture/sample* payload is
  a stretch goal, explicitly behind no network — see §11.)
- **No change to detection, routing, tone, confidence, risk, or recommendation.**
  Import only changes where threads *originate*; downstream behavior is invariant.
- **No frontend work required.** TypeScript types are optional and only added if a
  preview UI lands in a later phase (see §6).

## 3. Why this phase is fixture-only and network-free

- **Stable target, isolated risk.** The `CommentThread` contract is settled
  (Phases 14–16). Mapping pure JSON → contract is the riskiest *correctness* work
  (threading, file/line, resolved state); doing it network-free lets us nail it with
  deterministic, reviewable fixtures.
- **No security surface yet.** Tokens, SSRF, rate limits, and private-repo handling
  are real work that shouldn't gate (or be rushed alongside) the mapping logic.
- **Fast, hermetic tests.** Recorded fixtures make tests deterministic, offline, and
  immune to provider rate limits or schema drift during CI.
- **Honest about uncertainty.** We can encode our current understanding of provider
  JSON and flag exactly what must be re-verified against official docs before any
  live call (see §10).

## 4. Proposed backend models

New module, e.g. `backend/app/models/git_import.py`, using the existing `CamelModel`
base (snake_case fields, camelCase JSON). Names mirror the design doc.

```python
class GitProviderType(str, Enum):
    GITHUB = "github"
    GITLAB = "gitlab"


class ExternalCommentReference(CamelModel):
    """Provider-native identity preserved for provenance, dedupe, and (future)
    posting. Never affects review behavior."""
    provider: GitProviderType
    project: str | None = None                 # "owner/repo" or GitLab project path/id
    pull_request_number: int | None = None     # GitHub PR number
    merge_request_iid: int | None = None        # GitLab MR iid
    discussion_id: str | None = None           # GitLab discussion id
    review_id: str | None = None               # GitHub review id (if part of a review)
    comment_id: str | None = None              # provider comment/note id (thread root)
    web_url: str | None = None                 # human-openable link
    kind: str | None = None                    # "review_comment" | "issue_comment" | "discussion_note"


class ImportedCommentThread(CamelModel):
    """A normalized thread plus provenance. `thread` is the *existing* contract type
    consumed by the review/reply pipeline; `external` is optional side-car metadata."""
    thread: CommentThread
    external: ExternalCommentReference


class ImportCommentsRequest(CamelModel):
    """Inputs for a (future) import. In this phase the mappers take parsed JSON
    directly; this model documents the intended request surface and lets a stretch-
    goal endpoint accept a fixture-shaped payload. No URL fetching happens here."""
    provider: GitProviderType
    project: str | None = None
    pull_request_number: int | None = None
    merge_request_iid: int | None = None
    include_resolved: bool = True
    include_outdated: bool = False
    max_threads: int | None = None
    # Raw, already-fetched provider payload(s) — supplied by the caller/fixtures,
    # NOT fetched by us in this phase.
    raw_review_comments: list[dict] | None = None
    raw_issue_comments: list[dict] | None = None
    raw_discussions: list[dict] | None = None


class ImportCommentsResponse(CamelModel):
    """Result of normalization: contract-shaped threads + provenance + warnings."""
    threads: list[ImportedCommentThread]
    skipped: int = 0            # threads/comments dropped (empty body, no comments)
    warnings: list[str] = []    # structured, non-fatal normalization notes
```

> Note: `raw_*` fields are typed as `list[dict]` deliberately — we do **not** model
> provider schemas as strict Pydantic types in this phase, because those schemas are
> exactly what we still need to verify against official docs (§10). The mappers do
> defensive `.get(...)` access and emit warnings rather than hard-failing.

## 5. Proposed mapper functions (pure)

New package, e.g. `backend/app/services/git_import/` with one module per provider and
a small shared helpers module. **All functions are pure**: parsed JSON in, domain
objects out; no I/O, no globals, deterministic.

```python
# app/services/git_import/github.py
def map_github_review_comments_to_threads(
    comments: list[dict],
    *,
    project: str | None = None,
    pull_request_number: int | None = None,
    include_resolved: bool = True,
    include_outdated: bool = False,
) -> list[ImportedCommentThread]: ...

def map_github_issue_comments_to_threads(
    comments: list[dict],
    *,
    project: str | None = None,
    pull_request_number: int | None = None,
) -> list[ImportedCommentThread]: ...

# app/services/git_import/gitlab.py
def map_gitlab_discussions_to_threads(
    discussions: list[dict],
    *,
    project: str | None = None,
    merge_request_iid: int | None = None,
    include_resolved: bool = True,
    include_outdated: bool = False,
) -> list[ImportedCommentThread]: ...

# app/services/git_import/common.py  (shared helpers)
def synthetic_thread_id(ref: ExternalCommentReference) -> str: ...
def clean_body(raw: str | None) -> str | None: ...          # trim; None if empty
def to_thread_comment(...) -> ThreadComment | None: ...      # None if body empty
def resolve_status(*, resolved: bool | None, outdated: bool | None) -> CommentThreadStatus: ...
```

A thin orchestrator can combine them for an end-to-end fixture test:

```python
# app/services/git_import/__init__.py
def import_comments(req: ImportCommentsRequest) -> ImportCommentsResponse: ...
```

## 6. Proposed TypeScript types (optional, deferred to a UI phase)

No frontend work is required for this phase. If/when a preview UI is added later, the
frontend would mirror `ExternalCommentReference` and `ImportedCommentThread` in
`frontend/src/types/review.ts`. The existing `CommentThread` / `ThreadComment` /
`CommentThreadStatus` types already cover the normalized output, so the review/reply
UI needs no change. **This phase adds no TS types.**

## 7. Proposed fixture locations

Small, hand-curated, **synthetic** fixtures (no real users, repos, or tokens) under:

- `backend/tests/fixtures/github_pr_review_comments.json`
- `backend/tests/fixtures/github_pr_issue_comments.json`
- `backend/tests/fixtures/gitlab_mr_discussions.json`

Each fixture should exercise the tricky cases (threaded replies, line-less comments,
resolved vs. unresolved, outdated/missing line, empty/whitespace body to be dropped).
Fixtures are illustrative approximations of provider shapes and are explicitly marked
as **subject to verification against official docs** (§10).

### 7.1 GitHub fixture shapes to support

- **PR review comments** (`github_pr_review_comments.json`): line-anchored comments
  with fields like `id`, `user.login`, `body`, `created_at`, `path`, `line`,
  `original_line`, `in_reply_to_id`, `pull_request_review_id`, `html_url`. Reply
  chains are linked via `in_reply_to_id`; the fixture must include at least one root
  + reply to prove thread reconstruction. (GraphQL `PullRequestReviewThread.isResolved`
  is noted as the resolution source to verify later — REST may not expose it.)
- **PR issue comments** (`github_pr_issue_comments.json`): conversation-level comments
  with `id`, `user.login`, `body`, `created_at`, `html_url` and **no** `path`/`line`.

### 7.2 GitLab fixture shapes to support

- **MR discussions** (`gitlab_mr_discussions.json`): a list of discussions, each with
  `id` and ordered `notes[]`. Each note has `id`, `author.username`, `body`,
  `created_at`, `system` (filter out system notes), `resolvable`, `resolved`, and a
  `position` object (`new_path`, `old_path`, `new_line`, `old_line`) for positional
  notes. Include a non-positional (MR-level) discussion and a resolved one.

## 8. Normalization rules (mapping table)

How provider data maps onto the existing contract (see
[`review-contract.md`](review-contract.md) for field definitions):

| Target | Source / Rule |
| --- | --- |
| `ExternalCommentReference.*` | Provider ids captured verbatim: GitHub `id`→`comment_id`, `pull_request_review_id`→`review_id`, `html_url`→`web_url`; GitLab discussion `id`→`discussion_id`, note `id`→`comment_id`. `project`/PR#/MR-iid come from the call args. `kind` records the source kind. |
| `CommentThread.id` | **Synthetic, deterministic** from `ExternalCommentReference` (e.g. `github:owner/repo#42:rc:<rootCommentId>`, `gitlab:proj!7:disc:<discussionId>`) so re-imports are idempotent. Native ids live in `external`. |
| `ThreadComment.body` | Provider `body`, trimmed. Empty/whitespace-only → comment **dropped**; a thread with no remaining comments is **skipped** (contract requires ≥1, bodies non-empty). |
| `ThreadComment.author` | GitHub `user.login` / GitLab `author.username` (display only; `null` if absent). |
| `ThreadComment.id` | Provider comment/note id, stringified. |
| `ThreadComment.createdAt` | Provider `created_at`, passed through as a string when present. |
| `CommentThread.filePath` | GitHub `path` / GitLab `position.new_path` (fallback `old_path`). `null` for issue/MR-level comments. |
| `CommentThread.line` | New-side line preferred (GitHub `line`/`original_line`; GitLab `position.new_line`), old-side fallback. **Outdated** (line no longer present) → `filePath` kept, `line = null`. `null` for line-less comments. |
| `CommentThread.status` | See §8.1. |
| `CommentThread.source` | `"github"` / `"gitlab"`. |

### 8.1 Status mapping (`CommentThreadStatus` = `open` / `resolved` / `unknown`)

The existing enum has **no** `outdated` value, so:

- provider reports **resolved** → `resolved`.
- provider reports **unresolved/open** → `open`.
- provider exposes **no** resolution state → `unknown`.
- **outdated but unresolved** → `open`, with `line` nulled (file kept) and a `warnings`
  note; outdatedness is provenance, not a status value.
- `include_resolved=False` filters out resolved threads; `include_outdated=False`
  filters out outdated ones (default keeps resolved, drops outdated per §4).

### 8.2 Issue-level / non-positional comments → line-less threads

GitHub PR **issue comments** and GitLab **non-positional discussions** have no
file/line. Each becomes a `CommentThread` with `filePath = null`, `line = null`.
GitHub issue comments have no native threading, so (absent a future grouping
heuristic) each issue comment maps to its own single-comment thread.

## 9. Test strategy (fixture-based, offline)

All tests load the checked-in fixtures and assert pure-mapper behavior — **no network,
no live calls**.

- **GitHub review comments:** reply chains reconstructed into the correct threads
  (root + replies, chronological); `filePath`/`line` populated; ids land in
  `ExternalCommentReference`; synthetic thread id is stable.
- **GitHub issue comments:** each → a single line-less thread; `filePath`/`line` null.
- **GitLab discussions:** discussion→thread, notes→comments in order; system notes
  filtered; positional file/line mapped (new-side preferred, old-side fallback);
  resolved/unresolved → status; non-positional → line-less.
- **Edge cases:** empty/whitespace bodies dropped; empty threads skipped; outdated →
  line nulled + warning; missing fields tolerated (defensive `.get`, warning emitted).
- **Idempotency:** mapping the same fixture twice yields identical thread ids and
  output.
- **Contract conformance:** every produced `thread` is a valid `CommentThread`
  (passes existing validators) and can be fed into `run_review` unchanged.
- **Invariance:** running a review on imported threads vs. the same threads as local
  input yields **identical** suggested replies, routing, confidence, findings, risk,
  and recommendation. (Locks in "import is provenance/presentation only.")

No fixtures or tests are added in *this* (planning) pass — they're written in the
implementation phase.

## 10. Known uncertainties to verify against official docs before any live API work

These are encoded as assumptions in fixtures/mappers and **must be re-checked against
official GitHub/GitLab API docs before adding live calls**:

- **GitHub thread resolution.** Whether resolved state is reliably available via REST
  vs. only GraphQL `PullRequestReviewThread.isResolved`. Fixtures should not assume a
  REST field exists until verified.
- **GitHub review-comment threading.** Exact semantics of `in_reply_to_id` vs.
  GraphQL review threads, and how multi-level replies are represented.
- **GitHub line anchoring.** Current meaning of `line`, `original_line`,
  `start_line`, `side`, `position`, and how "outdated" is signaled.
- **GitLab position object.** Exact fields for positional notes (`new_line`,
  `old_line`, `line_range` for multi-line) and how outdated/resolvable is reported.
- **GitLab system notes.** The reliable way to identify and exclude system notes
  (the `system` flag) across versions.
- **Pagination & shapes.** Real payloads are paginated and may differ from our
  synthetic fixtures; field presence/casing must be confirmed.
- **Self-hosted differences.** GitLab self-managed / GitHub Enterprise may differ.

The plan deliberately keeps provider schemas as `dict` (not strict models) so these
unknowns don't force premature, possibly-wrong type definitions.

## 11. Security / privacy guardrails (even though network-free)

- **Synthetic fixtures only.** No real tokens, secrets, private repo content, emails,
  or PII in committed fixtures.
- **No secrets in code or fixtures.** Nothing token-like is introduced this phase.
- **Treat bodies as untrusted text.** Mappers don't execute or interpret comment
  bodies; downstream rendering already treats them as text.
- **Author data minimization.** Capture only a display author; do not import emails or
  extra profile data.
- **Forward-looking only.** SSRF guard, host allowlist, token redaction, and
  rate-limit/backoff handling are documented in
  [`future-git-provider-comment-import.md`](future-git-provider-comment-import.md)
  and are **not** implemented here (there is nothing to fetch).
- **Stretch-goal endpoint stays local.** If a thin `POST /api/import-comments` is
  added, it accepts a fixture-shaped payload and calls the pure mappers — it performs
  **no** outbound network I/O.

## 12. Suggested implementation phases

1. **Contracts (models only).** ✅ *Done.* Added
   [`backend/app/models/git_import.py`](../backend/app/models/git_import.py) with
   `GitProviderType`, `ExternalCommentReference`, `ImportedCommentThread`,
   `ImportCommentsRequest`, `ImportCommentsResponse` (exported from
   `app/models/__init__.py`) and contract tests in
   `backend/tests/test_git_import_contracts.py`. No mappers, fixtures, network,
   tokens, endpoints, or UI. `ImportCommentsRequest` carries **no token** and
   triggers **no** fetch — a future phase passes already-fetched JSON via
   `raw_payload`. Provider-native identity lives in `ExternalCommentReference`
   (`repository`/`project_id`, PR#/MR-iid, `discussion_id`/`review_id`/`comment_id`/
   `note_id`, `web_url`, `is_outdated`), keeping the core `CommentThread` untouched.
2. **Shared helpers + GitHub review-comment mapper.** ✅ *Done.* Added
   `app/services/git_import/{__init__,common,github}.py`: pure helpers
   (`clean_body`, `synthetic_thread_id`, `to_thread_comment`, `resolve_status`,
   `first_present`) and `map_github_review_comments_to_threads(...)`. The mapper
   consumes already-parsed dicts (defensive `.get`), reconstructs root/reply chains
   via `in_reply_to_id`, maps `path`/`line` (current preferred, original fallback),
   preserves provenance in `ExternalCommentReference` (repository, PR#, review id,
   comment id, web url, outdated flag), drops empty-body comments, skips
   outdated/resolved threads per `include_outdated`/`include_resolved` (outdated, when
   included, nulls the line + warns), and surfaces missing-root replies as standalone
   threads with a warning. Output is deterministic (first-seen root order; readable
   ids like `github:acme/widgets:7:rc:1001`). Fixture
   `backend/tests/fixtures/github_pr_review_comments.json` (synthetic) + tests in
   `backend/tests/test_github_review_comment_mapper.py` cover all of the above. No
   network, tokens, endpoints, or UI.
3. **GitHub issue-comment mapper.** ✅ *Done.* Added
   `map_github_issue_comments_to_threads(...)` to `app/services/git_import/github.py`
   (reusing the shared helpers). PR conversation (issue) comments are line-less and
   unthreaded, so each non-empty comment becomes its own single-comment
   `CommentThread` (`filePath=None`, `line=None`, status `unknown`), with provenance
   in `ExternalCommentReference` (provider, repository, PR#, comment id, web url;
   `reviewId`/`discussionId`/`isOutdated` stay `None`). Empty bodies dropped; output
   follows input order; ids are deterministic (`github:…:ic:<id>`). Synthetic fixture
   `backend/tests/fixtures/github_pr_issue_comments.json` + tests in
   `backend/tests/test_github_issue_comment_mapper.py`. Network-free, no tokens, no
   endpoint.
4. **GitLab discussions mapper.** ✅ *Done.* Added
   `app/services/git_import/gitlab.py` with `map_gitlab_discussions_to_threads(...)`
   (reusing the shared helpers). One discussion → one `CommentThread`; ordered
   non-system, non-empty `notes[]` → ordered `ThreadComment`s. System notes
   (`system: true`) are filtered; empty bodies dropped; a discussion with no usable
   notes is dropped. Positional notes map `position.new_path`/`new_line` (fallback
   `old_path`/`old_line`); non-positional discussions are line-less. Resolution is
   derived from notes' `resolvable`/`resolved` (all-resolved → `resolved`, some
   unresolved → `open`, none → `unknown`), honoring `include_resolved`; outdated
   positions are skipped by default and, when `include_outdated=True`, the line is
   nulled + a warning recorded. Provenance lands in `ExternalCommentReference`
   (provider gitlab, `projectId`, `mergeRequestIid`, `discussionId`, first usable
   `noteId`, `webUrl`, `isOutdated`). Deterministic ids `gitlab:…:disc:<id>`.
   Synthetic fixture `backend/tests/fixtures/gitlab_mr_discussions.json` + tests in
   `backend/tests/test_gitlab_discussion_mapper.py`. Network-free, no tokens, no
   endpoint.
5. **Orchestrator + invariance test.** ✅ *Done.* Added
   `app/services/git_import/orchestrator.py` with the pure
   `import_comments(request: ImportCommentsRequest) -> ImportCommentsResponse`. It
   dispatches by `source` (`github_review_comments`, `github_issue_comments`,
   `gitlab_discussions`) — validating the source matches the provider — and infers the
   source only when unambiguous (GitLab has one mapper); ambiguous GitHub or
   unsupported combinations raise a clear `ValueError`. Empty `rawPayload` returns an
   empty response with a single warning; dict payloads can nest the item list and
   optional repo/PR/project/MR metadata; per-thread warnings are kept intact and also
   surfaced at the top level. Tests in `backend/tests/test_git_import_orchestrator.py`
   cover dispatch, empty/ambiguous/unsupported handling, warning surfacing,
   determinism, and model conformance — plus **invariance**: GitHub- and GitLab-
   imported threads drive `run_review` byte-identically to hand-authored local
   `commentThreads` (same findings, risk, recommendation, diff stats, and suggested
   replies). The optional local-only `POST /api/import-comments` endpoint was **not**
   added this phase (kept endpoint-free per scope).
6. **Docs sync.** Update `review-contract.md` (import types), `architecture.md`
   (where the mapper layer sits), and `decisions.md` as implementation lands.

## 13. What remains explicitly deferred

- **Live GitHub/GitLab API calls** (the mappers only consume parsed JSON).
- **OAuth, token input, private-repo support** — a separate, later auth/security phase.
- **Auto-posting comments and posting suggested replies** — replies stay copy-only,
  `needsHumanReview: true`; posting (if ever) is a gated phase after import with auth,
  preview, auditability, and dedupe safeguards.
- **Real AI/LLM generation** — reply generation remains deterministic and local.
- **Frontend preview UI / TypeScript import types** — optional, future.
