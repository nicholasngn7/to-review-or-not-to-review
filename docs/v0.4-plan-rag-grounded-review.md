# v0.4 plan — RAG-style grounded review context

> **Status: planned, not implemented.** This document is a design/implementation plan
> only. As of this writing there is **no** ingestion, embedding, retrieval, or citation
> code in the repo, **no** new dependencies, and **no** tests for these features. The
> shipped product remains: a deterministic mock review provider behind a provider
> abstraction, a normalized `CommentThread` contract, a local fixture-based comment
> import demo, and copy-only suggested replies.

## 1. Product goal

Add **RAG-style local retrieval** to MR Review Council so a review can be *grounded* in
the project's own written context — repository documentation, architecture notes, and
(optionally) the MR/diff text itself. Before a review runs, the system retrieves the
most relevant context chunks and makes them available to the review provider; relevant
findings can then carry **citations** (a chunk id plus a short snippet) so a reader can
see *what context informed a point*.

The first implementation is deliberately a **RAG architecture demo**: it proves the
end-to-end pipeline — ingest → chunk → embed → retrieve → cite — using a **deterministic
local embedding provider** and no external services, mirroring how v0.1–v0.3 proved
review/import flows behind seams before taking on cloud cost or vendor lock-in.

## 2. Non-goals

- Not a production-grade or production-quality RAG system.
- No semantic/neural embedding model in the first cut (the local provider is
  deterministic and lexical, not a learned model).
- No live/Bedrock/OpenAI embedding calls in v0.4 unless a real provider path is
  explicitly implemented and tested in a later phase.
- No LLM-generated review findings — findings remain produced by the deterministic mock
  provider; retrieval only *grounds* and *annotates* them.
- No autonomous/agentic behavior, multi-step planning, or tool use.
- No vector database or external index service; the index is in-memory/local.
- No crawling of external sites or arbitrary URLs; ingestion is limited to local,
  caller-selected files.
- No persistence of indexes across processes in the first cut (rebuild on demand).

## 3. Accurate positioning

Use this wording:

- “RAG-style local retrieval” / “RAG architecture demo”.
- “deterministic local embedding provider”.
- “provider-ready retrieval pipeline” (the embedding step sits behind an abstraction so
  a real model can be added later).
- “lexical/deterministic similarity” for the first provider.

Do **not** use, unless and until actually implemented and tested:

- “production-grade RAG” / “production RAG”.
- “semantic search quality” / “semantic retrieval” (before a learned embedding model
  exists).
- “Bedrock-powered” / “LLM-powered” / “AI-generated review findings”.
- “agentic AI reviewer” / “autonomous agent”.

Honest framing: *v0.4 demonstrates the retrieval architecture and grounds the review UI
with citations, using a deterministic local embedding provider. Retrieval quality is
lexical, not semantic, until a real embedding provider is implemented.*

## 4. Architecture overview

The retrieval pipeline is a new, self-contained module that the review flow calls
**before** producing findings. It reuses the project's existing patterns: Pydantic
contracts (`CamelModel`), a provider abstraction selected by env var, pure/deterministic
services, and fixture-based tests.

```text
                ┌────────────────────────── knowledge pipeline (new) ───────────────────────────┐
 local files →  │ ingestion → chunking → EmbeddingProvider.embed() → in-memory KnowledgeIndex    │
 (md/docs)      │                              ▲                                                 │
                │            EMBEDDING_PROVIDER selects: LocalDeterministic (default)             │
                └──────────────────────────────┼─────────────────────────────────────────────────┘
                                                │ retrieve(top_k) by cosine similarity
 review request ───► review_engine ──► RetrievalQuery ──► RetrievalResult (chunks + scores)
                          │                                        │
                          ▼                                        ▼
                    ReviewProvider.review(... , retrieved_context) ─► findings (+ optional citations)
                          │
                          ▼
                    ReviewResponse (+ context_used: RetrievedCitation[])
```

Key boundaries:

- **EmbeddingProvider** abstraction mirrors `ReviewProvider`: one `embed(texts)` method,
  selected by an `EMBEDDING_PROVIDER` env var via a validating factory. Default is a
  deterministic local provider; a `bedrock`/`openai` provider is an optional later seam
  that returns a clear error unless configured.
- **Retrieval is additive and optional.** When no knowledge sources are provided, the
  review behaves exactly as today (invariance requirement — see §11).
- **Citations are provenance, not behavior.** Like `ExternalCommentReference` in v0.3,
  citations annotate output; they must not change risk scoring, severities, or the merge
  recommendation.

New backend layout (proposed):

```text
backend/app/models/knowledge.py            # contracts (Phase 1)
backend/app/services/knowledge/
  __init__.py
  ingestion.py                             # file selection + read (Phase 2)
  chunking.py                              # heading/paragraph/code-block chunker (Phase 2)
  embeddings/
    __init__.py
    base.py                                # EmbeddingProvider interface (Phase 3)
    local_provider.py                      # deterministic local provider (Phase 3)
    factory.py                             # create_embedding_provider(EMBEDDING_PROVIDER)
  index.py                                 # in-memory KnowledgeIndex + cosine top-k (Phase 3)
  retrieval.py                             # ingest()/search() service functions (Phase 4)
backend/tests/fixtures/knowledge/          # eval docs + cases (Phase 6)
```

## 5. Proposed backend contracts (Phase 1)

All in `backend/app/models/knowledge.py`, `CamelModel`-based (snake_case Python →
camelCase JSON), additive to existing models. **Contracts only in Phase 1** — no
behavior.

- **`EmbeddingProviderType(str, Enum)`** — `LOCAL_DETERMINISTIC = "local-deterministic"`
  (default). Reserve names like `BEDROCK = "bedrock"`, `OPENAI = "openai"` for future
  work; they are enum values only here, with no implementation.

- **`KnowledgeDocument`** — a selected source file before chunking.
  - `id: str` (stable; e.g. slug of the repo-relative path)
  - `source_path: str` (repo-relative path, e.g. `docs/architecture.md`)
  - `title: Optional[str]` (first H1 or filename)
  - `content: str` (raw text)
  - `content_hash: str` (stable hash of `content`, for change detection/determinism)
  - `metadata: dict[str, str]` (e.g. `{"kind": "doc"}`)

- **`KnowledgeChunk`** — a retrievable unit.
  - `id: str` (stable: `f"{document_id}#{ordinal}"` or `document_id` + heading slug)
  - `document_id: str`
  - `ordinal: int` (0-based position within the document)
  - `text: str`
  - `heading_path: list[str]` (e.g. `["Architecture overview", "Key boundaries"]`)
  - `kind: ChunkKind` (`heading_section` | `paragraph` | `code_block`)
  - `source_path: str`, `start_line: Optional[int]`, `end_line: Optional[int]`
  - `token_estimate: Optional[int]`

- **`EmbeddingVector`** — provider-tagged vector.
  - `provider: EmbeddingProviderType`
  - `dimension: int`
  - `values: list[float]`
  - `model_id: Optional[str]` (e.g. `"local-deterministic-v1"`)

- **`RetrievalQuery`** — what to retrieve for.
  - `text: str` (query text; built from MR title/description/diff or a free query)
  - `top_k: int = 5` (`ge=1`, `le=50`)
  - `document_ids: Optional[list[str]]` (optional scope filter)
  - `min_score: Optional[float]` (`ge=0.0`, `le=1.0`; optional cutoff)

- **`RetrievedCitation`** — a single grounded snippet (provenance).
  - `chunk_id: str`, `document_id: str`, `source_path: str`
  - `heading_path: list[str]`
  - `snippet: str` (short, trimmed excerpt for display)
  - `score: float` (`ge=0.0`, `le=1.0`; similarity score, provider-relative)

- **`RetrievalResult`** — the retrieval output.
  - `query: RetrievalQuery`
  - `provider: EmbeddingProviderType`
  - `citations: list[RetrievedCitation]` (ranked, length ≤ `top_k`)
  - `warnings: list[str]`

- **`RetrievalEvaluationCase`** — one fixture eval case (Phase 6).
  - `id: str`
  - `query: str`
  - `expected_chunk_ids: list[str]` (gold top-k, order-insensitive set for scoring)
  - `top_k: int`
  - `notes: Optional[str]`

Additive changes to existing review contracts (proposed, **optional/back-compatible**):

- `ReviewRequest`: add `knowledge_sources: Optional[list[str]]` (repo-relative paths to
  ingest for this review) and `retrieval: Optional[RetrievalQuery]` (override defaults).
  Absent ⇒ retrieval is skipped and behavior is unchanged.
- `ReviewFinding`: add `citations: list[RetrievedCitation] = []` (default empty;
  populated only when a finding is grounded).
- `ReviewResponse`: add `context_used: list[RetrievedCitation] = []` (the distinct
  context surfaced for this review; empty when retrieval did not run).

## 6. Proposed frontend changes (Phase 7)

TypeScript types mirror the new contracts in `frontend/src/types/` (e.g.
`knowledge.ts`). UI is additive and honest:

- A small, optional **“Knowledge sources”** control in the input panel to pick which
  bundled/local docs to ground on (default: a curated set, e.g. `docs/architecture.md`,
  `README.md`). No URL or upload-from-web input.
- A **“Context used”** section in the review results showing the retrieved citations
  (source path, heading path, snippet, score), collapsible like the tone/threads panels.
- On finding cards, an optional **“Grounded in”** affordance listing that finding's
  citations (snippet + source path), copy-friendly.
- Honest microcopy: label it “RAG-style local retrieval (deterministic)”, and state that
  snippets come from local project docs, not an external search or an LLM.
- Markdown export gains an optional “Context used” block when citations exist.

## 7. Proposed local embedding provider design (Phase 3)

`EmbeddingProvider` (interface, `base.py`):

```python
embed(texts: list[str]) -> list[EmbeddingVector]   # batch, order-preserving, pure
```

`LocalDeterministicEmbeddingProvider` (`local_provider.py`):

- **Deterministic and offline.** No network, no credentials, no randomness without a
  fixed seed. Same input text ⇒ identical vector every run.
- **Lexical hashing approach (hashing vectorizer / bag-of-features):**
  - Normalize text (lowercase, strip punctuation, split on non-word boundaries; optional
    light stemming/stopword list kept in-repo and version-controlled).
  - Map each token to a bucket via a stable hash (e.g. `blake2`/`sha1` of the token mod
    `dimension`), accumulate counts (optionally hashed bigrams for a little context).
  - Apply sublinear term weighting (e.g. `1 + log(count)`) and **L2-normalize** so
    cosine similarity is well-behaved.
  - Fixed `dimension` (e.g. 512 or 1024) and a `model_id` like `"local-deterministic-v1"`
    so vectors are versioned.
- **No learned semantics.** This captures lexical overlap only — documented plainly as
  such. It is enough to prove the pipeline, ranking, and citations deterministically.

`KnowledgeIndex` (`index.py`):

- Holds chunks + their `EmbeddingVector`s in memory.
- `search(query_vector, top_k, min_score)` computes **cosine similarity** (vectors are
  pre-normalized, so this is a dot product), sorts descending, applies `top_k`/`min_score`,
  and returns ranked `RetrievedCitation`s. Ties broken by stable chunk id ordering for
  determinism.

`create_embedding_provider(EMBEDDING_PROVIDER)` (`factory.py`): validating factory,
default `local-deterministic`; unknown values raise a clear `ValueError`. A future
`bedrock`/`openai` value maps to a provider that errors unless configured (see §14).

## 8. Proposed retrieval flow (Phase 4 + 5)

Local-only service functions (and optionally thin endpoints), no external calls:

- `ingest_knowledge(paths: list[str]) -> KnowledgeIndex` — read selected local files,
  chunk them, embed chunks, build the in-memory index. Deterministic given the same
  files.
- `search_knowledge(index, query: RetrievalQuery) -> RetrievalResult` — embed the query
  text with the same provider, run `index.search`, return ranked citations.

Optional endpoints (local-only, mirroring `/api/import-comments`):

- `POST /api/knowledge/ingest` — body lists repo-relative paths (server-side allow-list
  to the repo's own `docs/`/`README.md`); returns document/chunk counts. No URL fetching.
- `POST /api/knowledge/search` — body is a `RetrievalQuery`; returns a `RetrievalResult`.

Review integration (Phase 5), in `review_engine`:

1. If `ReviewRequest.knowledge_sources` is empty/absent → **skip retrieval entirely**;
   produce the review exactly as today (no `context_used`, no citations).
2. Otherwise ingest the selected sources (or reuse a cached index keyed by file hashes),
   build a `RetrievalQuery` from the MR title/description/diff (or the provided
   `retrieval` override), and run `search_knowledge`.
3. Pass the `RetrievalResult` into `ReviewProvider.review(..., retrieved_context=...)`.
   The deterministic mock provider attaches a finding's relevant citations when a chunk
   clearly relates (e.g. shared file path or strong lexical overlap) — **without changing
   which findings exist, their severity, or the verdict**.
4. Populate `ReviewResponse.context_used` with the distinct citations surfaced.

## 9. Proposed citation model

Covered by `RetrievedCitation` (§5). Principles:

- **Provenance only.** Citations never alter risk, severity, finding existence, or the
  merge recommendation (enforced by invariance tests, §11). They are the retrieval
  analogue of v0.3's `ExternalCommentReference`.
- **Stable ids.** `chunk_id` is deterministic and re-derivable from the source, so
  evaluation fixtures can assert exact ids.
- **Display-ready snippets.** `snippet` is a trimmed excerpt (bounded length, whitespace-
  collapsed) suitable for a card; `heading_path` gives human context; `score` is shown as
  provider-relative similarity, explicitly *not* a confidence or correctness measure.
- **Attachment points.** Per-finding `citations` (grounding a specific point) and
  response-level `context_used` (everything surfaced). A finding with no relevant context
  simply has an empty list.

## 10. Proposed evaluation fixture strategy (Phase 6)

Make retrieval quality *measurable and deterministic* without a learned model:

- **Fixed corpus.** A small, version-controlled doc set under
  `backend/tests/fixtures/knowledge/` (synthetic or copies of selected real docs), so
  chunk ids are stable.
- **Fixed cases.** `RetrievalEvaluationCase` fixtures (JSON) pairing a `query` with the
  `expected_chunk_ids` a correct lexical retriever should return for a given `top_k`.
- **Metrics (lexical, honest).** Compute set-based **precision@k**, **recall@k**, and
  **hit@k** (did any gold chunk appear) over the cases. Assert thresholds the
  deterministic provider can actually meet; treat the suite as a regression guard, not a
  claim of semantic quality.
- **Determinism check.** Re-running ingestion/embedding/search yields identical chunk
  ids, vectors, and rankings.
- **Eval harness.** A small pure function `evaluate(index, cases) -> EvaluationReport`
  (counts + per-case hits) usable from tests and a future CLI; no network.

## 11. Testing strategy

(Tests are written during implementation, not in this planning phase.)

- **Contracts (Phase 1):** model validation, camelCase serialization round-trips,
  enum values, optional/default fields, back-compatibility of the additive review fields.
- **Chunking (Phase 2):** deterministic chunk ids/ordinals; correct splitting on
  headings/paragraphs/code blocks; `heading_path`/line ranges; idempotence.
- **Embedding provider (Phase 3):** determinism (identical vectors across runs), fixed
  dimension, L2-normalization, cosine sanity (identical text → 1.0; disjoint vocab → ~0),
  and top-k ordering with stable tie-breaking.
- **Retrieval service (Phase 4):** ingest/search happy paths, `top_k`/`min_score`/scope
  filters, empty corpus and empty query handling, warnings.
- **Review integration invariance (Phase 5 — critical):** for the same diff, a review
  **with no knowledge sources** is byte-for-byte identical to today's output (no
  `context_used`, no citations); and adding knowledge sources **does not change**
  findings, severities, overall risk, or the merge recommendation — it only adds
  `citations`/`context_used`. This is the v0.4 analogue of v0.3's import-invariance tests.
- **Evaluation (Phase 6):** precision@k/recall@k/hit@k thresholds on the fixture cases;
  determinism of the eval report.
- **Frontend (Phase 7):** “Context used” renders when citations exist and is absent
  otherwise; finding-card grounding renders; export includes context when present.

## 12. Demo flow

A repeatable, fully local walkthrough (no credentials, no external network):

1. Load a demo diff (e.g. the risky backend auth change).
2. In **Knowledge sources**, select bundled local docs (default set pre-selected).
3. Run Review. Retrieval runs first against the deterministic local index.
4. Review the results: open the **“Context used”** section to see ranked citations
   (source path, heading path, snippet, score), and note the **“Grounded in”** snippets
   on relevant finding cards.
5. (Optional) Export Markdown — the report includes the “Context used” block.
6. Re-run to show determinism (identical retrieval and rankings).

Exact-version screenshots/videos would be captured with the existing Playwright harness
once implemented (see `frontend/demo/README.md`), framed honestly as a “RAG architecture
demo with a deterministic local embedding provider.”

## 13. Risks and mitigations

| Risk | Mitigation |
| ---- | ---------- |
| Overclaiming retrieval quality | Label it lexical/deterministic; reserve “semantic” for a real embedding model; keep §3 wording rules in docs and UI copy. |
| Retrieval subtly changing the verdict | Enforce invariance tests (§11): citations are provenance only; no path may alter findings/severity/risk/recommendation. |
| Non-determinism (hashing, ordering, floats) | Fixed hash + seed, L2-normalized vectors, stable tie-breaking by chunk id, fixed `dimension`/`model_id`; determinism tests. |
| Reading arbitrary files / path traversal | Server-side allow-list to the repo's own docs; no URL/web ingestion; reject paths outside the allowed roots. |
| Index cost on large repos | Bound corpus to selected files; in-memory only; document scope; cache by file hash. |
| Scope creep toward live providers | Keep embedding provider behind a factory; ship only the local provider; Bedrock/OpenAI remain disabled design (§14). |
| Snippet leakage of sensitive content | Ingest only chosen, local, non-secret docs; snippets are bounded excerpts of those files. |
| New dependencies | Prefer stdlib (`hashlib`, `math`) for the local provider; defer heavier libs until/unless a real provider is implemented. |

## 14. Future Bedrock / provider path (Phase 8, optional)

- Add a real `EmbeddingProvider` implementation (e.g. `BedrockEmbeddingProvider`,
  `OpenAIEmbeddingProvider`) behind the **same** `embed()` interface and factory.
- **Disabled by default.** Selected only when `EMBEDDING_PROVIDER` is set to that value
  **and** the required configuration/credentials are present; otherwise the factory
  returns a provider that fails with a clear, explanatory error (mirroring the review
  `bedrock` placeholder's `501`) — never a silent fallback or a fabricated vector.
- Vectors stay provider-tagged (`EmbeddingVector.provider`/`model_id`) so an index built
  with one provider isn't mixed with another.
- Only after such a provider is implemented **and tested** may docs describe semantic
  retrieval or name the vendor (e.g. “Bedrock-backed embeddings”). Until then it is
  documented strictly as an optional future path.

## 15. Accuracy boundaries / out of scope

For v0.4 as planned:

- **Deterministic, lexical retrieval — not semantic.** The default embedding provider is
  a deterministic local hashing vectorizer; it does not understand meaning.
- **No real AI/LLM.** Review findings remain from the deterministic mock provider;
  retrieval grounds and annotates, it does not generate findings.
- **No live/Bedrock embedding calls** unless Phase 8 is explicitly implemented and tested.
- **Not production-grade RAG.** It is a local RAG **architecture demo** and provider-ready
  pipeline.
- **No autonomous/agentic behavior.**
- **Local-only ingestion.** Only caller-selected files within the repo's allowed doc
  roots; no web crawling, no URL fetching, no external index service, no token/OAuth.
- **No cross-process persistence** of the index in the first cut.

## Suggested build sequence (phases)

1. **Phase 1 — Contracts only.** `KnowledgeDocument`, `KnowledgeChunk`,
   `EmbeddingProviderType`, `EmbeddingVector`, `RetrievalQuery`, `RetrievalResult`,
   `RetrievedCitation`, `RetrievalEvaluationCase` (+ additive optional review fields).
   No behavior.
2. **Phase 2 — Ingestion + chunking.** Local file selection (allow-listed), chunk by
   headings/paragraphs/code blocks, stable ids, deterministic metadata.
3. **Phase 3 — Deterministic local embedding provider.** Hashing vectorizer, cosine
   similarity, top-k retrieval, in-memory index; unit tests for determinism/ranking.
4. **Phase 4 — Retrieval API.** Local-only `ingest`/`search` service functions (and
   optional local endpoints); no external/paid calls.
5. **Phase 5 — Review integration.** Retrieve before review, pass context to the
   deterministic provider, attach citations where relevant, preserve existing behavior
   when no knowledge context exists (invariance tests).
6. **Phase 6 — Evaluation fixtures.** Fixed docs, fixed queries, expected top-k chunk
   ids, precision@k/recall@k/hit@k regression tests.
7. **Phase 7 — Frontend.** “Context used” section and per-finding grounding; honest copy;
   optional export block.
8. **Phase 8 — Optional provider path.** Real embedding provider behind the factory,
   disabled unless configured; documented as optional future work.

Each phase is additive and independently testable, consistent with the v0.1–v0.3 build
style. See [`decisions.md`](decisions.md) for the recorded v0.4 (planned) decision and
[`project-case-study.md`](project-case-study.md) for where this sits in the roadmap.
